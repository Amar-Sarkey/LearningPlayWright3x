# Source Code vs Byte Code vs Binary Code

## Example Source Code

```javascript
console.log("Hello, World!");
```

This is our source file: `01_HelloWorld.js`

---

## Comparison Table

| Aspect | Source Code | Byte Code (Intermediate) | Binary Code (Machine Code) |
|--------|-------------|--------------------------|----------------------------|
| **Definition** | Human-readable instructions written in a programming language | Intermediate representation between source and machine code | Direct instructions executed by the CPU |
| **Who reads it** | Humans (developers) | Virtual Machine (e.g., V8, JVM, .NET CLR) | Physical CPU processor |
| **Readability** | Highly readable (`console.log("Hello, World!")`) | Partially readable (opcodes + operands) | Not human-readable (raw bits) |
| **Example content** | `console.log("Hello, World!");` | `get_property r0, "log"`<br>`push "Hello, World!"`<br>`call r0, 1` | `10101000 01001111 ...` (hex: `48 8D 0D ...`) |
| **Portability** | Highly portable — runs anywhere with the right runtime | Portable across platforms that have the same VM | Tied to a specific CPU architecture (x86, ARM, etc.) |
| **Generated by** | Written by the developer in a text editor | Compiler (e.g., V8 Ignition, javac, Roslyn) | Assembler or JIT compiler (e.g., V8 TurboFan) |
| **Execution model** | **Interpreted** line-by-line (slowest) | **Interpreted or JIT-compiled** (faster) | **Executed directly** by CPU (fastest) |
| **Optimization level** | None (raw developer intent) | Basic optimizations applied | Highly optimized for the target CPU |
| **File size** | Smallest (1 line = ~33 bytes) | Medium | Largest |
| **Security** | Fully visible and modifiable | Reverse-engineerable but harder to modify | Very hard to understand or tamper with |

## How They Relate (Using Our Example)

### 1. Source Code

```
console.log("Hello, World!");
```

This is what the developer writes. It's plain text, easy to read, and expresses *intent* rather than step-by-step CPU instructions. JavaScript engines like V8 (Chrome, Node.js) cannot execute this directly.

### 2. Byte Code (V8 Ignition)

When Node.js runs this file, V8's **Ignition interpreter** first parses the source into **byte code**. A simplified version might look like:

```
LdaConstant  r0, "Hello, World!"    // Load the string constant
GetProperty  r1, global, "console"  // Get the console object
GetProperty  r2, r1, "log"          // Get the log method
Call         r2, r0                  // Call log with the string
Return                               // Return
```

Byte code is **platform-independent** — the same byte code works on x64 Windows, ARM macOS, or any other architecture that has a V8 VM. It's more compact than source but not yet ready for the CPU.

### 3. Binary Code (V8 TurboFan JIT)

If the code runs frequently (becomes "hot"), V8's **TurboFan** optimizing compiler converts the byte code into **native machine code** for the specific CPU:

```asm
; x86-64 assembly representation
mov   rcx, qword [rsi + 0x18]      ; Load console object
mov   rax, qword [rcx + 0x20]      ; Load log method
lea   rdx, [string "Hello, World!"] ; Load string address
call  rax                           ; Invoke the function
```

Which the CPU sees as raw bytes:

```
48 8B 4E 18 48 8B 41 20 48 8D 15 ...
```

This is **binary code** — the only thing the CPU natively understands.

## Key Takeaways

| Code Type | Speed | Human Readable | Platform Independent | Example |
|-----------|-------|----------------|---------------------|---------|
| **Source** | Slowest (needs translation) | ✅ Yes | ✅ Yes | `console.log("Hello, World!");` |
| **Byte** | Medium (VM interprets) | ⚠️ Partial | ✅ Yes | `LdaConstant r0, "Hello, World!"` |
| **Binary** | Fastest (direct CPU) | ❌ No | ❌ No | `48 8B 4E 18 48 8B 41 20` |

### Why JavaScript Uses This Multi-Stage Approach

JavaScript is **JIT-compiled** (Just-In-Time), not purely interpreted nor purely compiled. This gives the best of both worlds:
- **Fast startup**: Ignition starts executing byte code immediately (no long compile phase)
- **Fast long-running performance**: TurboFan optimizes hot paths into native binary code on the fly
- **Portability**: Byte code runs anywhere V8 exists
